# Sample Pictures Picker

A popover-based picker that lets users run the existing pipeline on the
project's `sample-pictures/` directory without having to upload anything.
Selections persist in localStorage; on first load (no stored entry) every
sample picture is preselected so a new visitor can hit "Process" once and
see results.

## Goals

- One-click access to the sample images from inside the running web app.
- Sample-picture assets follow the same build-time copy + generated-manifest
  pattern as `frames` and `instructions-assets`, so adding a new file to
  `sample-pictures/` requires no code changes.
- Selection state survives reloads via `useLocalStorage`.
- The popover always shows its Submit button (no scroll-to-find).
- Reuse the existing `ImageInput.onImagesSelected` callback so the
  archive-then-process flow in `App.handleImagesSelected` runs unchanged.

## Non-goals

- Thumbnail pre-generation. The originals are copied verbatim and the
  browser scales them in `<img>` tags (matches `copy-frames.ts`). Can be
  revisited if the sample set grows large.
- Per-picture metadata (captions, descriptions, ordering hints). Filenames
  drive both display order (alphabetical) and identity.
- Editing/uploading new sample pictures from the UI. Adding samples is a
  repo-level operation: drop a file into `sample-pictures/` and rebuild.
- Migrating selections from old filenames if a sample is renamed; stale
  entries are silently dropped on read.

## Build step

### New script: `packages/gbcam-extract-web/scripts/copy-sample-pictures.ts`

Mirrors `copy-frames.ts` exactly in shape:

- **Source:** `<projectRoot>/sample-pictures` (i.e. `../../../sample-pictures`
  relative to the script).
- **Destination (assets):** `packages/gbcam-extract-web/public/sample-pictures/`.
- **Destination (manifest):**
  `packages/gbcam-extract-web/src/generated/SamplePictures.ts`.
- **Whitelisted extensions:** `.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`
  (same set as `copy-frames.ts`).
- **Order:** alphabetical by filename, so the popover's display order is
  deterministic and matches what's on disk.
- **Empty-source behavior:** if `sample-pictures/` does not exist, log a
  warning and write a manifest with an empty array — same fallback shape
  as `copy-frames.ts`. The component then hides its trigger.

The generated module:

```ts
// auto-generated by scripts/copy-sample-pictures.ts — do not edit
export interface SamplePictureEntry {
  /** URL the browser fetches (relative to the deployed root). */
  url: string;
  /** Source filename including extension, e.g. "20260313_213416.jpg". */
  filename: string;
}

export const SAMPLE_PICTURES: ReadonlyArray<SamplePictureEntry> = [
  /* ... */
];
```

URLs are written as `./sample-pictures/<filename>` to match the
relative-path convention used by `copy-frames.ts`.

Add the new per-asset script next to `build:frames` plus a combined
`build:assets` script that runs all three asset builders. Replace the
repeated `build:instructions && build:frames` chains in `dev`, `dev:host`,
`build`, `preview`, and `preview:host` with a single call to
`pnpm build:assets`. Update `postinstall` to use it too (with
`generate-licenses` still chained on, since that's specific to install):

```jsonc
{
  "scripts": {
    "build:instructions": "node scripts/generate-instructions.ts",
    "build:frames": "node scripts/copy-frames.ts",
    "build:sample-pictures": "node scripts/copy-sample-pictures.ts",
    "build:assets": "pnpm build:instructions && pnpm build:frames && pnpm build:sample-pictures",
    "dev": "pnpm build:assets && vite",
    "dev:host": "pnpm build:assets && vite --host",
    "build": "pnpm build:assets && tsc -b && vite build",
    "preview": "pnpm build:assets && vite preview",
    "preview:host": "pnpm build:assets && vite preview --host",
    "postinstall": "pnpm build:assets && node scripts/generate-licenses.ts"
  }
}
```

Adding a future asset builder is now a one-line change to `build:assets`
instead of a sweep across six scripts.

### Render shape

Hidden entirely (returns `null`) when `SAMPLE_PICTURES.length === 0`, so
the button row stays clean during local dev without sample pictures.

Otherwise:

- **Trigger:** `<PopoverTrigger render={<Button variant="secondary"
  disabled={disabled}>...} >Sample Pictures<ChevronDown data-icon="inline-end" /></PopoverTrigger>`,
  matching the base-ui composition pattern used in `App.tsx` (`Dialog`,
  `Collapsible`).
- **Content:** `<PopoverContent
  className="flex flex-col w-[min(28rem,90vw)] max-h-[min(80vh,32rem)] gap-2"
  align="start" side="bottom">`.
- **Layout inside content** (top to bottom):
  1. `PopoverHeader` with `PopoverTitle` "Sample pictures" and a
     `PopoverDescription` showing the running count, e.g.
     `"3 of 5 selected"`.
  2. Scrollable grid container `<div className="flex-1 overflow-y-auto">`
     holding `<div className="grid grid-cols-2 gap-2">`. Each tile is a
     `<button>` toggle:
     - Wrapper: `relative rounded-md border bg-muted p-1 ring-2
       ring-transparent transition-colors`. When selected:
       `ring-primary border-primary`.
     - Image: `<img src={entry.url} loading="lazy" className="block w-full
       h-24 object-contain" alt="" />`. (`object-contain` keeps tall
       phone-portrait photos and landscape photos visible without
       cropping; `h-24` keeps the popover compact at ~96px per row.)
     - Filename: `<span className="block text-xs text-muted-foreground
       truncate mt-1">{entry.filename}</span>` for orientation.
     - Selected check overlay: a `Check` icon from `lucide-react`
       absolutely positioned at the top-end corner inside a small
       `bg-primary text-primary-foreground rounded-full size-5` badge,
       only rendered when selected.
     - `aria-pressed={isSelected}` and `aria-label={\`Toggle
       ${entry.filename}\`}`.
  3. Footer `<div className="flex items-center justify-end gap-2 pt-2
     border-t">` holding the Submit button. Sitting outside the scroll
     container guarantees it's always visible.
- **Submit button:** `<Button onClick={handleSubmit} disabled={isSubmitting
  || effectiveSelected.length === 0}>{isSubmitting ? "Loading…" :
  \`Process ${n} picture${n === 1 ? "" : "s"}\`}</Button>`.

### State

- `const [open, setOpen] = useState(false)`.
- `const [storedSelection, setStoredSelection] =
  useLocalStorage<string[] | null>("gbcam-sample-picture-selections",
  null)`. Stored value is `string[] | null` so we can distinguish "never
  set" (use the all-selected default) from "set to []" (genuinely empty
  selection).
- `const validFilenames = useMemo(() => new
  Set(SAMPLE_PICTURES.map((s) => s.filename)), [])`.
- `const effectiveSelected = useMemo(() => storedSelection === null ?
  SAMPLE_PICTURES.map((s) => s.filename) :
  storedSelection.filter((f) => validFilenames.has(f)),
  [storedSelection])`.
- `const [isSubmitting, setIsSubmitting] = useState(false)`.

`storedSelection === null` means "user has never made a choice on this
device" → default to all selected. Once the user toggles anything, we
write a concrete array (possibly `[]`); after that the array — including
empty — is authoritative regardless of what gets added to
`sample-pictures/` later.

### Toggle handler

```ts
const toggle = (filename: string) => {
  setStoredSelection((prev) => {
    const base = prev === null
      ? SAMPLE_PICTURES.map((s) => s.filename)
      : prev.filter((f) => validFilenames.has(f));
    return base.includes(filename)
      ? base.filter((f) => f !== filename)
      : [...base, filename];
  });
};
```

The first toggle "materializes" the implicit all-selected default into a
concrete array and then applies the toggle, so subsequent reads no
longer rely on the null branch.

### Submit handler

```ts
const handleSubmit = async () => {
  const selectedEntries = SAMPLE_PICTURES.filter((s) =>
    effectiveSelected.includes(s.filename),
  );
  if (selectedEntries.length === 0) return;

  setIsSubmitting(true);
  try {
    const settled = await Promise.allSettled(
      selectedEntries.map(async (entry) => {
        const res = await fetch(entry.url);
        if (!res.ok) {
          throw new Error(`Failed to fetch ${entry.filename}: ${res.status}`);
        }
        const blob = await res.blob();
        const type = blob.type || guessMimeFromName(entry.filename);
        return new File([blob], entry.filename, { type });
      }),
    );

    const files: File[] = [];
    const errors: string[] = [];
    for (const result of settled) {
      if (result.status === "fulfilled") files.push(result.value);
      else errors.push(result.reason instanceof Error
        ? result.reason.message
        : String(result.reason));
    }

    if (errors.length > 0) {
      toast.error(`Failed to load ${errors.length} sample picture${
        errors.length === 1 ? "" : "s"
      }`, { description: errors.join("\n") });
    }

    if (files.length > 0) {
      onImagesSelected(files);
      setOpen(false);
    }
  } finally {
    setIsSubmitting(false);
  }
};
```

`guessMimeFromName` is a small inline helper covering the same extension
set the build script accepts — defensive fallback for browsers/servers
that report empty `blob.type`. Selection state is **not** mutated by
submit; the next open shows the same checks.

`toast` comes from `sonner`, already mounted at the top of `App.tsx`.

### Accessibility

- Each tile is a real `<button>` with `aria-pressed`.
- Submit button is a real `<Button>`; disabled state communicated via
  `disabled` (works with shadcn's primitives).
- Keyboard: tiles tabbable, Space/Enter toggles. Esc closes the popover
  (handled by base-ui).

## Wiring into `ImageInput.tsx`

Inside the existing button row (`<div className="flex gap-3 mt-3">`),
mount the picker after the Camera Capture button:

```tsx
<div className="flex gap-3 mt-3">
  <Button variant="secondary" ...>Choose Files</Button>
  <Button variant="secondary" ...>Camera Capture</Button>
  <SamplePicturePicker
    onImagesSelected={onImagesSelected}
    disabled={disabled}
  />
</div>
```

No changes to `App.tsx`. `handleImagesSelected` already archives current
results to history and runs `processFiles`, so submitting from the picker
behaves identically to a fresh upload.
