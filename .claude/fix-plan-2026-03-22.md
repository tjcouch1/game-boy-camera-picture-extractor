## Problem 1 (CRITICAL): zelda-poster-3 — 471 errors, 3.29%

### Root Cause: G-valley search range is binding at its constraint boundary

**Error pattern:** 421 LG->WH errors (pipeline says LG, reference says WH),
concentrated in the **bottom-right corner** (cols 97-127, rows 75-111).
A further 17 are BK->DG and 16 are WH->LG.

**Mechanics of LG->WH:**
The G-valley threshold classifies high-R pixels as:

- G >= threshold -> WH (label 3)
- G < threshold -> LG (label 2)

LG->WH error = pipeline output LG = pixel has G < threshold.
Reference says WH -> pixel truly belongs in WH.
Fix needed: **lower the threshold** so pixels with G in [new_threshold, 220) flip to WH.

**Why threshold is stuck at 220:**
In `_g_valley_threshold()`, the valley search is restricted to the upper 2/3 of
the `[lg_center_g, wh_center_g]` range. For zelda-poster-3:

LG centre = 171.2, WH centre = 244.6
lo = 172, hi = 244 -> 73 histogram bins
search_lo = 73 \* 2 // 3 = 48 -> search starts at G = 172 + 48 = 220

The minimum of smooth[48:] is at index 0 -- smooth[48] is already the lowest value
in the restricted range. Threshold snaps to exactly 220. **The constraint is binding.**
The actual valley (likely around G = 200-210) is in the excluded lower portion.

**Confirmed binding images (valley_idx == search_lo for every one):**

| Test           | LG ctr | WH ctr | bins | search_lo | Threshold | Status  |
| -------------- | ------ | ------ | ---- | --------- | --------- | ------- |
| zelda-poster-3 | 171.2  | 244.6  | 73   | 48        | 220       | BINDING |
| thing-2        | 164.8  | 240.6  | 76   | 50        | 215       | BINDING |
| zelda-poster-2 | 156.5  | 245.9  | 90   | 59        | 216       | BINDING |

**Non-binding images (valley found interior to range -- must NOT regress):**

| Test           | search_lo -> G_start | Found threshold | Status      |
| -------------- | -------------------- | --------------- | ----------- |
| thing-1        | 48 -> G=226          | 237             | not binding |
| thing-3        | 54 -> G=221          | 227             | not binding |
| zelda-poster-1 | 56 -> G=221          | 225             | not binding |

The proposed fix only activates when `valley_idx == search_lo`. For the three
non-binding images the condition is false and behavior is **identical** to current code.

### Fix 1A -- Adaptive G-valley search range (gbcam_quantize.py)

**File:** `gbcam_quantize.py`
**Function:** `_g_valley_threshold()`
**Location:** Block immediately after `gaussian_filter1d()`, approx lines 85-92.

**Current code:**

```python
smooth = gaussian_filter1d(hist.astype(float), sigma=3.0)
# Only look for valley in the upper half of the range (near WH) to avoid
# picking up the dense LG body as the minimum
search_lo = len(smooth) * 2 // 3
valley_idx = search_lo + int(np.argmin(smooth[search_lo:]))
threshold = float(edges[valley_idx])
```

**New code:**

```python
smooth = gaussian_filter1d(hist.astype(float), sigma=3.0)
# Start search from upper 2/3 of range to avoid the dense LG body.
search_lo = len(smooth) * 2 // 3
valley_idx = search_lo + int(np.argmin(smooth[search_lo:]))

# If the search was boundary-constrained (minimum is at search_lo itself),
# the true valley lies lower in the range than the restrictive window allows.
# Retry from the lower 1/3 point -- this still stays well above the LG body
# peak (near LG_centre) while admitting valleys in the middle of the range.
if valley_idx == search_lo:
    wider_lo = max(len(smooth) // 3, 1)
    valley_idx = wider_lo + int(np.argmin(smooth[wider_lo:]))

threshold = float(edges[valley_idx])
```

**Expected impact:**

- zelda-poster-3: threshold drops 220 -> ~200-210. The 421 mis-classified WH pixels
  (G in ~200-219) flip to WH. Estimated: ~400 fewer errors, net ~60-70 remaining.
- thing-2: threshold may drop slightly from 215. Errors there are LG<->DG-dominated
  (not LG/WH), so limited benefit but also no expected regression.
- zelda-poster-2: threshold may drop from 216, potentially fixing 1-3 of 6 errors.
- thing-1, thing-3, zelda-poster-1: **no change** (not binding -> condition is false).
